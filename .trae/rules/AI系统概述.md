## 一、AI配置管理器（核心中枢）

[`AI配置管理器.cs`](file:///d:/D/淼喵妙脚本DIY应用程序/淼喵妙DIY应用程序/淼喵妙神奇工具库/AI配置管理器.cs)（约 1706 行）是整个 AI 系统的中枢，负责所有 LLM 调用。

### 三套 AI 配置

系统支持多套独立的命名配置，不再只有单一全局配置：

| 配置角色 | 用途 | 获取方法 |
|----------|------|----------|
| **文本AI配置** | 对话聊天的默认 LLM | `获取全局配置()` → 从配置列表第一项读取 |
| **视觉AI配置** | 截图分析的多模态 AI（如 GPT-4o、Qwen-VL） | `获取视觉AI配置()` → 按 `视觉AI配置Id` 查找 |
| **评审AI配置** | 计划评审的独立 LLM（可用更强模型） | `获取评审AI配置()` → 按 `评审AI配置Id` 查找 |

每个对话还独立记录 `上次文本配置Id` 和 `上次多模态配置Id`，新建对话时自动继承上次使用的配置。

### 两种 AI 提供者

| 提供者类型 | 字符串值 | 底层实现 |
|-----------|----------|----------|
| **Ollama 本地** | `"Ollama本地"` | `OllamaApiClient` (OllamaSharp)，向本地 Ollama 服务发请求 |
| **OpenAI 兼容 API** | `"OpenAI 兼容 API"` | `HttpClient` 直接 POST SSE 流式解析，支持 Bearer 令牌认证 |

### 核心方法一览

#### 文本对话

| 方法 | 说明 |
|------|------|
| `调用AI(AIConfigData, 消息历史, 自定义规则)` | 同步非流式调用，分派到本地或远程 |
| `调用AI流式(AIConfigData, 消息历史, 规则, 回调, 思考回调?, cancToken)` | 流式调用入口，委托给 `调用AI流式带工具` |
| `调用AI流式带工具(config, 消息历史, 规则, 工具列表, 回调, 工具回调?, 思考回调?, 进度回调?, cancToken, 当前计划树?)` | **核心流式方法**：发送消息 → 流式接收 → 计划提取 → 计划评审 → 工具调用 → 计划验证 → 递归 |
| `调用本地AI流式带工具(...)` | Ollama 流式生成 + `流式思考解析器` 分离思考/正文 + `<tool_call>` 标签解析 + 计划评审 + 递归工具执行 |
| `调用远程AI流式带工具(...)` | HttpClient SSE 流式解析 + `reasoning_content` 思考分流 + `MCP工具管理器.构建工具定义列表()` 生成 tools 数组 + 最大 5 轮工具调用循环 + 计划验证 |

#### 多模态视觉

| 方法 | 说明 |
|------|------|
| `捕获屏幕截图()` | 截取主屏幕并编码为 `data:image/png;base64,...` 格式 |
| `调用AI分析截图(config, base64Image, 提示词)` | 多模态分析入口，分派到本地或远程 |
| `调用本地AI分析截图(...)` | Ollama `GenerateRequest.Images` 字段传入 base64，默认模型 `minicpm-v:latest` |
| `调用远程AI分析截图(...)` | OpenAI Vision API 格式（`image_url` content），支持 DashScope 千问多模态（`/api/v1/services/aigc/multimodal-generation/generation`） |

#### 系统提示词构建

| 方法 | 说明 |
|------|------|
| `构建提示词(消息历史, 规则)` | Ollama 格式：拼接经验系统消息、计划要求、规则、通知解读指南、MCP工具指南、历史消息 |
| `构建消息列表(消息历史, 规则)` | OpenAI `messages` 数组：含 `reasoning_content`、`image_url` 多模态、`system` 多层提示词 |
| `收集分类规则提示词(工具列表)` | 从所有工具的 `分类AI规则` 和 `祖先AI规则列表` 构建层级分类树，注入系统提示 |
| `构建经验系统消息从历史(消息历史)` | 从消息历史提取已调用过的工具ID列表，构建 AI使用经验管理器的上下文 |

#### 配置管理

### 持久化

- **主配置文件**：`%AppData%/淼喵妙脚本DIY/ai_config.json`
- 内容：AI 配置列表（多命名）、全局自定义规则、全局快捷指令列表、所有对话（含消息历史）、上次文本/多模态配置Id、视觉/评审AI配置Id、启用自主学习/增量记录/过滤私有消息等开关
- 序列化：`System.Text.Json`（注意：旧版用 Newtonsoft.Json，已迁移）

### 数据迁移

静态构造函数中调用 `执行数据迁移()`：将旧的单一 `全局AI配置` 迁移到 `AI配置列表` 中的 `AINamedConfig`，确保向后兼容。

### 重要约定

- **全局配置**：通过 `AI配置管理器.获取全局配置()` 获取（取配置列表第一项），用于没有对话上下文的场景（如 `智能规划任务` 节点）
- **对话级配置**：每个 `AIConversation` 通过 `文本AI配置Id` 和 `多模态AI配置Id` 引用独立的命名配置
- **默认规则**：全局规则默认为 `"始终用中文回复。如任务需要多个步骤，请先简要列出计划，再逐一调用工具执行。"`
- **线程安全**：所有写操作通过 `lock(_锁)` 保护；流式调用使用 `AsyncLocal` 传递上下文
- **`.ConfigureAwait(false)`**：所有 `await` 必须使用 `.ConfigureAwait(false)`，确保 SSE 循环在 ThreadPool 上运行而非 UI 线程

### AIChatMessage 关键特性

- **思考内容分离**：AI 回复中的推理过程（`reasoning_content`）存储在独立的 `思考内容` 字段，UI 层可折叠/展开
- **私有消息**：`是否私有=true` 的系统消息（如工具执行结果）不会出现在后续对话的上下文中（当 `过滤私有消息=true` 时）
- **图片列表**：用户消息可附带多个 base64 编码的图片，构建消息时转为 `image_url` content parts

## 三、MCP工具管理器

[`MCP工具管理器.cs`](file:///d:/D/淼喵妙脚本DIY应用程序/淼喵妙DIY应用程序/淼喵妙神奇工具库/MCP工具管理器.cs)（约 401 行）将脚本包装为 AI 可调用的工具。

### 数据模型

| 字段 | 类型 | 说明 |
|------|------|------|
| `名称` | `string` | 原始中文脚本名 |
| `工具ID` | `string` | **ASCII 安全标识符**，用作 LLM function name（中文名 → 提取 ASCII 字符 → 无 ASCII 则 MD5 hash 兜底 `tool_xxxxxxxxxxxx`） |
| `描述` | `string` | 工具描述（含原始中文名） |
| `脚本路径` | `string` | `.script` 文件绝对路径 |
| `分类路径` | `string` | 脚本在分类层级中的完整路径（如 `指令与流程控制/脚本构建`） |
| `分类AI规则` | `string` | 该分类的 AI 使用规则（如"用于构建新的自动化脚本"） |
| `祖先AI规则列表` | `List<string>` | 所有父级分类的 AI 规则（递归继承） |
| `祖先分类路径列表` | `List<string>` | 与祖先规则对应的分类路径 |
| `参数Schema` | `object` | 可选参数 Schema |
| `已有使用经验` | `bool` | 该工具是否已有 AI 总结经验 |
| `经验更新时间` | `string` | 上次经验更新的时间戳 |
| `已有统计数据` | `bool` | 该工具是否有调用统计数据 |

### 工作流程

1. **加载工具列表**：根据对话的 `工具库分类列表`，从 `userdata.json` 中递归收集匹配分类下的 `.script` 文件
2. **递归收集**：`递归收集分类脚本()` 遍历分类树，子分类自动继承祖先分类的 AI 规则
3. **生成 ASCII 工具名**：`生成ASCII工具名(中文名)` → 提取 ASCII 字符 → 降级为 MD5 hash（`tool_` 前缀）
4. **去重处理**：同名工具自动追加 `_1`、`_2` 后缀
5. **经验状态查询**：加载后自动查询每个工具的 `已有使用经验` 和 `已有统计数据` 状态
6. **构建工具定义**：`构建工具定义列表()` 返回通用 function calling 格式（`name` 使用 `工具ID`，`description` 含分类标记和中文名）
7. **解析工具调用**：
   - Ollama：正则匹配 `<tool_call>工具ID</tool_call>` 或 `<tool_call>工具ID|{"参数":"值"}</tool_call>`
   - OpenAI：解析 `response.choices[].message.tool_calls`
8. **执行工具**：区分内置工具（`通用MCP工具.内置工具ID列表`）和脚本工具：
   - 内置工具 → `通用MCP工具.执行内置工具()`
   - 脚本工具 → 读取脚本文件 → 创建 `自动任务脚本` → `脚本.执行(进度回调)` → 返回结果

### 关键方法

| 方法 | 说明 |
|------|------|
| `生成ASCII工具名(string)` | 中文 → ASCII 安全标识符 |
| `加载工具列表(List<string>)` | 从持久化数据中递归加载指定分类的脚本（含祖先规则继承） |
| `递归收集分类脚本(...)` | 遍历分类树，匹配目标分类并收集脚本 |
| `构建工具定义列表(List<MCPToolDefinition>)` | 生成通用 function calling 格式的 tools 数组 |
| `构建Ollama工具描述()` | 生成 Ollama 格式的工具提示（支持带参数格式说明） |
| `解析ToolCall响应(string, List<MCPToolDefinition>, ...)` | 正则匹配 `<tool_call>` 标签，解析 JSON 参数，执行工具 |
| `执行工具(string 工具标识, List<MCPToolDefinition>, 参数?, ...)` | 按 `工具ID` 或 `名称` 匹配并执行脚本/内置工具，支持 `__隐藏进度` 参数 |
| `转换为McpTool(MCPToolDefinition)` | 转换为 `McpDotNet.Protocol.Types.Tool` 格式 |
| `导出Mcp工具列表(List<MCPToolDefinition>)` | 批量导出 McpDotNet 格式工具列表 |

### 脚本输出收集

通过 `AsyncLocal<StringBuilder> _输出收集器` 实现：
- `MCP工具管理器.是否在MCP上下文中` — 判断当前是否在 MCP 工具执行中
- `追加输出(文本)` — 脚本执行过程中的节点输出会被收集
- `获取收集器输出()` — 工具执行完毕后，收集到的输出作为 AI 回复的一部分返回

### 相关依赖

- `mcpdotnet` + `McpDotNet.Extensions.AI` — MCP 协议支持库（NuGet 已引用，为未来 MCP 协议标准化预留）

## 四、通用MCP工具

## 五、AI使用经验管理器

[`AI使用经验管理器.cs`](file:///d:/D/淼喵妙脚本DIY应用程序/淼喵妙DIY应用程序/淼喵妙神奇工具库/AI使用经验管理器.cs)（约 829 行）是 AI 自我学习进化的核心系统，负责经验积累、计划评审和工具调用统计。

### 数据模型

| 类 | 说明 |
|----|------|
| `计划节点` | 树形结构：`需求`（目标描述）+ `工具ID`（叶子节点的工具标识）+ `子步骤`（`List<计划节点>`，嵌套子步骤） |
| `工具调用记录` | 单次工具调用的完整记录：`工具ID`、`调用时间`、`输入参数`、`是否成功`、`耗时ms`、`输出摘要`、`对话ID`、`对话轮次`、`调用时计划树`（完整计划树序列化）、`调用后树修改`（工具执行后对计划的修改） |
| `工具经验条目` | 单个工具的 AI 总结的经验规则文本 |
| `经验存储根` | 持久化根对象：`经验词典`（`Dictionary<string, 工具经验条目>`）+ `评审模型名` |
| `统计存储根` | 统计持久化根对象：`调用记录列表`（`List<工具调用记录>`） |

### 持久化

| 文件 | 路径 | 内容 |
|------|------|------|
| `ai_experience.json` | `%AppData%/淼喵妙脚本DIY/` | 每个工具ID对应的AI总结经验文本 |
| `ai_tool_statistics.json` | `%AppData%/淼喵妙脚本DIY/` | 所有工具调用的统计记录列表 |

### 核心方法

| 方法 | 说明 |
|------|------|
| `从回复中提取计划(string)` | 用正则 `[计划]...[/计划]` 从 AI 回复中解析计划树（`需求=`、`步骤=描述\|工具ID` 或 `步骤=描述\|*\|子步骤1\|工具ID\|子步骤2\|工具ID`） |
| `构建经验系统消息(List<string>)` | 根据当前工具列表，从已有经验中构建系统提示消息，注入到对话上下文 |
| `追加调用记录(工具调用记录)` | 在 MCP工具管理器执行工具 时自动调用，记录每次工具调用详情 |
| `更新工具经验(string, string)` | 更新/新增某工具的使用经验规则 |
| `获取工具经验(string)` | 获取某工具的已有经验 |
| `获取经验状态(string)` | 查询某工具是否有经验和统计数据（返回三个 bool/string 元组值） |
| `查找历史相似案例(List<string>, int)` | 按工具ID列表从统计中匹配历史调用记录，用于计划评审 |
| `评审计划(string, List<string>)` | 调用评审AI配置（独立AI）对当前计划进行分析，含历史案例分析 |
| `触发经验总结(AIConversation, AIConfigData)` | 对话结束后，调用AI自动总结该对话的工具使用经验，保存到 `ai_experience.json` |
| `执行微调()` | 基于统计记录进行模型微调（目前为占位，待实现） |
| `清空工具经验(string)` / `清空所有经验()` | 清除指定/全部经验 |

### 计划评审流程

在 `调用AI流式带工具` 方法中的集成流程：

1. **提取计划**：AI 回复完成后，调用 `从回复中提取计划()` 解析 `[计划]...[/计划]` 块
2. **首次工具调用前评审**：仅在第一个 round（远程）或首次工具执行前（本地）触发
3. **历史案例匹配**：`查找历史相似案例()` 按工具ID列表匹配历史调用记录
4. **AI 评审**：`评审计划()` 使用独立的评审AI配置，发送计划 + 历史案例给评审AI
5. **建议注入**：评审建议作为系统消息追加到消息历史 → 重新调用 AI 让其调整计划

### 计划验证

当 `启用增量记录=true` 时：
- AI 必须在首次工具调用前提交 `[计划]...[/计划]`
- 每个工具调用前验证其 `工具ID` 是否在计划树的叶子节点中（`工具在计划内()`）
- 不在计划内的工具调用会被拦截，返回错误信息

### 触发经验总结

在 `MainWindowViewModel._autoSaveTimer` 回调中自动触发：检测对话中最近有 AI 回复且用户已 30 秒未输入 → 调用 `触发经验总结()` → AI 总结工具使用经验 → 持久化到 `ai_experience.json`。
