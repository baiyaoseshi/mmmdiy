# 本地优先 AI 架构 + 经验系统重构 Spec

## Why
当前经验注入机制（将规则化经验注入系统 Prompt）在实际使用中被证明不太靠谱——外来的 API 模型本身就足够强大，经验注入反而浪费 token 且可能干扰模型判断。取而代之的是采用"培养本地模型"的新模式。配置层面改为本地优先、API 兜底。

另一个关键问题：当前脚本身份由文件名派生，改名后工具ID 变化导致训练数据断裂。

此外，旧的规则化经验（`ai_experience.json`）是静态文本规则。取而代之的是**印象机制**：每个工具维护一份动态演化的"印象"——从命名+备注出发，首次使用生成"第一印象"，后续逐步修正，严重矛盾时覆盖。**印象是工具的公共属性，所有 AI 实例共享**，在构建工具定义时自动融入工具描述，帮助 AI 更准确地理解和选择工具。

## What Changes

### 移除
- 移除经验注入：不再在 system prompt 中注入经验消息
- 移除计划评审：`评审计划()`、`评审AI配置Id`、Phase A/B
- 移除规则化经验：`ai_experience.json` 及 `更新工具经验`/`清空工具经验`/`获取经验状态`/`构建经验系统消息`
- 移除旧微调：`执行微调()`/`导出微调数据()`/`微调后评审模型名`
- 移除 `触发经验总结`（产出为文本规则）
- 移除计划格式懒校验
- 移除 `MCPToolDefinition.已有使用经验`/`经验更新时间`

### 保留
- 保留工具调用记录（`追加调用记录`、`ai_tool_statistics.json`）
- 保留计划解析（`从回复中提取计划`、`解析计划`）
- 保留 `MCPToolDefinition.已有统计数据`

### 新增
- **本地优先配置**：文本/多模态/经验 AI 各自独立配置，本地优先、API 兜底
- **稳定脚本身份 GUID**：脚本头部持久 GUID，工具ID 由 GUID 派生，改名不影响
- **共享印象机制**：每个工具维护一份所有 AI 共享的"印象"——演变自命名+备注，首次使用生成第一印象，后续修正/覆盖。融入工具描述，AI 调用工具时自然可见
- **经验模型评估**：经验 AI 评估工具调用质量 + 印象一致性，选择性记录训练数据
- **训练数据管理** + **模型训练管线**

## Impact
- Affected specs: `ai-self-learning`（大规模重构）
- Affected code:
  - `AI配置模型.cs` — AIConfigData 新增 `是否优先本地`；AIPersistenceData 移除评审/微调，新增经验AI

  - `MCP工具管理器.cs` — 工具ID 改用 GUID 派生；构建工具描述时融入印象
  - `AIChatPage.xaml.cs` — 移除经验总结计数

  - `AIConversationSettings.xaml.cs` + `.xaml` — 适配新配置

## ADDED Requirements

### Requirement: 本地优先 AI 配置架构
系统 SHALL 支持每个 AI 角色（文本、多模态、经验）独立配置本地和远程，本地优先、API 兜底。经验 AI 仅本地。

#### Scenario: 本地可用 → 本地
- **WHEN** 配置中 Ollama 模型名和地址均已填写
- **THEN** 调用本地 Ollama

#### Scenario: 本地不可用 → API 降级
- **WHEN** Ollama 模型名为空，但远程 API 已配置
- **THEN** 调用远程 OpenAI 兼容 API

#### Scenario: 两者都不可用 → 报错
- **WHEN** 配置中既无本地也无远程
- **THEN** 返回错误提示

#### Scenario: 经验 AI 仅本地，未配置则跳过
- **WHEN** 经验 AI 未配置
- **THEN** 跳过评估和印象管理，仅写统计

### Requirement: 稳定脚本身份标识（GUID）
系统 SHALL 在脚本文件序列化头部持久化 GUID（`脚本Id[GUID]`），首次保存生成，永不改变。工具ID 由 `tool_` + GUID 前12位派生。改名后工具ID 不变。

#### Scenario: 新脚本首次保存 → 生成 GUID
#### Scenario: 已有脚本加载 → 解析 GUID 不改动
#### Scenario: 旧格式兼容 → 自动生成 GUID，下次保存持久化
#### Scenario: 改名后工具ID 不变 → 印象、训练数据、统计连续
#### Scenario: 删除重建同名 → 新 GUID，不混淆

---

### Requirement: 共享印象机制 — 核心概念
系统 SHALL 为每个工具维护一份**所有 AI 实例共享**的"印象"，存储在 `ai_tool_impressions.json`。印象是工具的公共属性，不是特定 AI 的私有知识。

印象的生命周期：
```
初始印象（名称+备注, 置信度0.3）
    ↓ 首次成功使用
第一印象（经验AI根据实际调用生成, 置信度0.7）← 永久基石
    ↓ 后续使用
  ┌─ 一致 → 置信度+0.02（上限0.95）
  ├─ 偏差 → 追加修正，第一印象不变
  └─ 严重矛盾 → 覆盖第一印象，置信度重置0.5
```

#### Scenario: 初始印象 — 来自命名与备注
- **WHEN** 工具首次加载且无印象记录
- **THEN** 初始印象 = `"工具 [名称]：[备注]"`，置信度 0.3
- **THEN** 此初始印象对**所有** AI 可见

#### Scenario: 第一印象 — 首次成功使用后生成
- **WHEN** 任何 AI 成功调用某工具且该工具仅有初始印象
- **THEN** 经验 AI 根据实际调用过程生成"第一印象"，覆盖初始印象，置信度 0.7
- **THEN** 此第一印象立即对**所有** AI 可见
- **THEN** 第一印象成为后续所有修正的基石

#### Scenario: 修正 — 一致 / 偏差
- **WHEN** 任何 AI 调用工具，经验 AI 评估与当前印象的关系
- **THEN** 一致 → 置信度 +0.02（上限 0.95）
- **THEN** 偏差 → 当前印象追加修正（如"也适用于XX场景"），第一印象不变
- **THEN** 修正后立即对**所有** AI 可见

#### Scenario: 覆盖 — 严重矛盾
- **WHEN** 调用结果与第一印象完全相反
- **THEN** 第一印象被覆盖，置信度重置 0.5，旧印象保留在修正历史

#### Scenario: 多个 AI 并发使用同一工具
- **WHEN** 文本 AI 和另一个对话的 AI 同时调用同一工具
- **THEN** 各自独立评估，印象更新线程安全（锁保护），后写入者基于最新状态判断

### Requirement: 共享印象 — 融入工具描述
系统 SHALL 在构建工具定义列表时，将工具的当前印象融入其 `描述` 字段，使 AI 在扫描可用工具时自然获取印象知识。

> **注意区分两种场景：**
>
> | 场景 | 工具描述内容 | 目的 |
> |------|-------------|------|
> | **推理时**（function calling） | 原始描述 + `【印象】印象内容` | 给当前 AI 完整上下文做工具选择 |
> | **训练时**（JSONL input） | 仅 `[印象内容]` | 训练模型，排除用户可编辑的不稳定信息 |

#### Scenario: 工具有印象时增强描述
- **WHEN** 构建工具定义列表，某工具有印象（置信度 ≥ 0.5）
- **THEN** 工具描述格式：`[原始描述]【印象】[当前印象]（置信度: X%）`
- **THEN** AI 无需额外 system prompt 即可理解工具的实际行为特征

#### Scenario: 工具仅有初始印象时不增强
- **WHEN** 工具仅有初始印象（置信度 0.3，未经实际验证）
- **THEN** 工具描述保持原始（名称+备注），不附加印象
- **THEN** 避免未验证的猜测误导 AI

#### Scenario: 工具无印象时正常描述
- **WHEN** 工具完全无印象记录
- **THEN** 使用原始描述，不受影响

#### Scenario: 文本 AI 通过印象选择工具
- **WHEN** 用户说"帮我看看屏幕上有什么"
- **THEN** 文本 AI 扫描工具列表，看到 `ask_vision_ai` 描述含 "【印象】截取当前屏幕并调用视觉AI分析。注意：仅截取主显示器。（置信度: 85%）"
- **THEN** AI 立即知道该工具只截主屏，若用户有双屏需求可提前告知

### Requirement: 印象数据模型
存储在 `ai_tool_impressions.json`，每工具一条记录，含修正历史。详见 spec 正文。

### Requirement: 经验模型评估工具调用
系统 SHALL 在每次工具调用完成后，由经验 AI 评估质量和印象一致性。

#### Scenario: 评估 Prompt 含当前印象
- **WHEN** 经验 AI 评估某次工具调用
- **THEN** Prompt 包含：计划上下文 + 工具名 + 原始描述 + **当前印象** + 输入参数 + 执行结果
- **THEN** 输出：`{"quality": "high"|"low", "impression_match": "一致"|"偏差"|"矛盾"|"首次使用", "reason": "...", "tags": [...]}`
- **THEN** 30 秒超时，异常不阻塞

#### Scenario: 首次成功使用 → 生成第一印象
#### Scenario: 已有印象 → 根据 match 类型更新印象
#### Scenario: 低质量调用 → 仅写统计，不碰印象
#### Scenario: 经验 AI 未配置 → 仅写统计

### Requirement: 训练数据管理
系统 SHALL 维护训练数据集 `ai_training_data.jsonl`。训练样本**基于现有 `工具调用记录` 格式扩展**，保留计划树等关键字段，新增印象相关内容。`input` 字段以印象为主体描述工具，不包含用户可编辑的名称和备注。

**为什么排除用户名称/备注**：用户可随意改名和修改备注，将其纳入训练数据会导致模型学习到不稳定的标识信息。印象由经验 AI 根据实际调用过程评估生成，反映工具的真实行为特征，是更可靠的训练信号。

#### Scenario: 训练样本格式（基于工具调用记录扩展）
- **WHEN** 经验模型评估某工具调用为 high 且写入训练数据
- **THEN** 基于 `工具调用记录` 构造 JSON 行：
  ```json
  {
    "instruction": "用户需求（计划树根节点需求描述）",
    "input": "工具印象: [当前印象] | 输入参数: {...} | 当前计划: [计划树扁平化描述]",
    "output": "执行结果摘要",
    "quality": "high",
    "impression_match": "一致",
    "工具GUID": "a1b2c3d4e5f6...",
    "印象版本": 3,
    "是否成功": true,
    "耗时ms": 1234,
    "调用时计划树": { "需求": "...", "子步骤": [...] },
    "调用后树修改": [{ "需求": "...", "子步骤": [...] }],
    "时间戳": "2026-05-29T15:30:00"
  }
  ```
- **THEN** `input` 中不包含工具名称、用户备注、分类路径
- **THEN** `调用时计划树` 和 `调用后树修改` 来自原始 `工具调用记录`，完整保留

#### Scenario: 首印象前的样本
- **WHEN** 工具仅有初始印象（置信度 0.3，名称+备注）时产生高质量调用
- **THEN** 暂不写入训练数据，等待第一印象生成后再决定
- **THEN** 避免将用户未验证的描述直接作为训练信号

#### Scenario: 数据清洗
- **WHEN** 用户触发数据清洗
- **THEN** 去重（按 instruction+input 哈希）、过滤过短/过长样本、标记孤立样本（工具 GUID 在当前工具集中不存在）

#### Scenario: 评测集构建
- **WHEN** 用户触发评测集构建
- **THEN** 80/20 随机划分训练集和评测集

### Requirement: 模型训练管线
系统 SHALL 提供 SFT/DPO/RL 三个阶段的接口和默认 TODO 占位实现。

#### Scenario: 触发 SFT 微调
- **WHEN** 训练数据达到阈值（默认 50 条）
- **THEN** 后台执行 SFT，使用训练数据创建增强版 Ollama 模型

#### Scenario: DPO / RL
- **WHEN** 上一阶段就绪且数据满足条件
- **THEN** 依次执行 DPO（偏好对数据）、RL（TODO 占位）

#### Scenario: 用户可覆盖训练逻辑
- **WHEN** 用户修改训练方法中 TODO 注释标记区域的代码
- **THEN** 下次触发训练时使用新逻辑

---

### Requirement: 训练触发时机
系统 SHALL 在窗口最小化到系统托盘 5 分钟后自动触发训练管线，也可在全局设置中手动触发。

#### Scenario: 最小化托盘 5 分钟后自动触发
- **WHEN** 窗口被最小化到托盘（`MainWindow.Hide()` 被调用）
- **THEN** 启动 5 分钟倒计时定时器
- **THEN** 若 5 分钟内窗口被恢复（`Show()`），取消倒计时
- **THEN** 若 5 分钟到期且窗口仍在托盘状态，自动触发训练管线
- **THEN** 训练流程：数据清洗 → 评测集构建 → SFT（若达阈值）→ DPO（若有偏好对）→ RL（若就绪）

#### Scenario: 训练期间 AI 锁定
- **WHEN** 训练管线正在执行
- **THEN** 设置全局 `是否训练中` 标志
- **THEN** 任何 AI 对话请求返回"模型训练中，请稍候再试"
- **THEN** 全局设置中显示训练进度（当前阶段、已用时间）
- **THEN** 训练完成后自动解除锁定

#### Scenario: 手动触发（全局设置）
- **WHEN** 用户在全局设置点击"开始训练"按钮
- **THEN** 立即触发训练管线，同时锁定 AI 对话、显示进度

#### Scenario: 训练数据不足时跳过 SFT
- **WHEN** 训练数据 < 50 条
- **THEN** 仅执行数据清洗和评测集构建，跳过模型微调
- **THEN** 日志中记录原因
